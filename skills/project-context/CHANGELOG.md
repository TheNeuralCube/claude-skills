# Changelog

All notable changes to the project-context skill will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this skill adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

The skill version (e.g., `0.5.0`) tracks releases. Beginning with v0.4.0, the **schema version** (`schema_version` in every file's frontmatter) is **decoupled** from the skill version. The schema version bumps only when the shape of stored data changes; the skill version bumps on every release. See `references/schema-changelog.md` for the version-by-version history of the data schema.

## [0.7.0] — 2026-06-09

This release consolidates the cancelled standalone v0.6.1 patch (the operation-start model advisory and the end-of-run set-integrity directive) and ships them at full fidelity alongside the schema-0.5 work, rather than as a generic micro-patch later replaced.

### Added
- **Versioned, prefix-unified file identity (schema 0.5).** The three managed context files move from fixed canonical names to `pc-NNNN-{context,entities,archive}.md`, where `NNNN` is a zero-padded four-digit shared-set counter recorded in frontmatter as the new REQUIRED `generation` field. The first generation is `pc-0001-*`. Recognition is by the `pc-NNNN-*` name pattern AND the `_managed_by` marker together. This resolves the duplicate-canonical-file corruption that fixed names produced on platforms that accumulate uploads instead of overwriting. Schema bumps "0.4" → "0.5".
- **`update_count` retained as the scoring counter (build finding F1).** The design's intent to retire `update_count` was reversed during the build: `update_count` remains the per-file scoring lifecycle counter that drives update-based decay, while `generation` is a separate identity-only counter. They are decoupled; `generation` resets to 1 on Scenario G, `update_count` is preserved verbatim. See `references/schema-changelog.md` schema 0.5 entry.
- **Counter assignment and the confirmed-empty rule** (`references/preflight.md` section 3.4). Confirmed-empty inventory seeds `0001`; a populated inventory increments to the highest observed counter plus one; an inventory that cannot be confirmed prompts the operator for a number higher than the highest they see. `0001` is never written on absence of evidence. Web reads the invocation manifest and never prompts; other surfaces are detect-and-react.
- **Generation self-consistency check** (`references/preflight.md` section 3.5). The filename `NNNN` must equal the frontmatter `generation`; a mismatch surfaces a diagnostic note.
- **Operation-start model advisory and two-tier confirmation gate** (`references/preflight.md` section 4.6; rolled from the cancelled v0.6.1, item 1). A non-blocking advisory names the platform's `strongest_thinking_model` (generic-wording fallback if missing). Routine operations (default, merge_external, regular compact) are advisory-only; destructive operations (rebuild, large compact, Scenario G migration) block on operator model-setup confirmation. The read-only audit trigger is gate-exempt.
- **Set-integrity directive** (`references/preflight.md` section 9.6; rolled from the cancelled v0.6.1, item 2). A single canonical directive, rendered unconditionally by the post-flight summary, names the full `pc-NNNN-*` set and instructs deletion of lower-numbered sets. The ad-hoc replace lines in the four operation files are replaced by a reference to this canonical directive. The change-aware download list and unchanged-file indicator are preserved.
- **config and references separation (convention P4).** Operator-editable files live in `config/` (`user-config.md`, `org-config.md`, `platform-specific-parameters.md`, shipped as `.template`); skill-owned files remain in `references/`. The skill reads config by base name, never a hardcoded `config/` path. New file `references/configure.md` owns the config interview mechanics once (batch, confirm, diff, write, compliance flag for governance fields); config files carry a self-documenting header (`config_editable`, `configure_with`) and a local field guide. A proposed cross-skill convention, co-proven with the session-handoff sister skill.
- **Expanded `config/platform-specific-parameters.md`.** Per platform: `strongest_thinking_model`, `can_change_active_model_from_skill`, `file_inventory_enumerable`, `config_read_location`, `output_bundling`, alongside the retained v0.6.0 instruction-rendering fields. Platforms: `claude-ai` (Claude Fable 5), `codex` (GPT-5.5), `chatgpt-enterprise` (GPT-5.5 Pro), and `m365-copilot` (conservative defaults pending validation: `file_inventory_enumerable: false`, base-name read, null model).
- **Scenario G migration (v0.6.0 → v0.7.0, schema 0.4 → 0.5).** Detects old canonical names at schema 0.4 with a topology block and no `pc-NNNN-*` files; verdict `⚠ Upgrade Available (v0.6.0 to v0.7.0)`; destructive-tier gate; token `confirm v0.7.0 upgrade`. Renames to `pc-0001-*`, adds `generation: 1`, bumps schema to 0.5, preserves `update_count` and all per-record counters verbatim, carries topology verbatim, applies the config treatment (header + relocation), and instructs deletion of old-named files. Migration is built and validated this cycle (`references/migration.md` section 11).

### Changed
- **Schema bumps "0.4" → "0.5".** Only delta on the three context files: `generation` added; `update_count` retained; filename versioned. Config files gain `config_editable` and `configure_with`. The topology block carries forward verbatim.
- **Operation write steps** now stamp `generation = N` on all three files (the set advances together) and write `pc-000N-*` names, while `update_count` increments only on files with content changes.
- **Platform support.** Codex profiled (operator-confirmation where self-inspection is unreliable); Claude Code remains a declined platform with a companion-skill redirect; **Cowork is deferred** (no Cowork profile this cycle; the anti-autonomous-decision guardrail must keep firing).

### Notes
- The cancelled v0.6.1 consolidation: the two operator-safety guardrails planned as a standalone v0.6.1 patch are designed here at full fidelity rather than shipped generic.

## [0.6.0] — 2026-05-22

### Added
- **Topology metadata as part of project identity (v0.6.0 first principle P1).** A project's topology relationship to other projects is now part of its active state. Every file the skill writes carries a `topology` block in frontmatter with five role values: `hub`, `spoke-dev`, `spoke-solution`, `standalone`, `unclassified`. Hub projects own a spoke inventory; spokes own a Hub reference and source-version stamp. The schema, role definitions, spoke inventory format, audit trigger semantics, hybrid topology rules, and validation rules live in `references/topology.md` (new file). The vision doc decision V1 (federal-state governance model) and V3 (hybrid topology rule Hub > Solution > Skill) lock the design.
- **`references/topology.md` (new file).** Authoritative specification for the topology metadata schema (universal across all projects), role definitions, spoke inventory format (Hub projects only), audit trigger semantics, hybrid topology rules (parent field), and validation rules (required-by-role fields, no-empty-fields principle, role/ISO-8601/declared_by validation).
- **Audit trigger phrase** invokable in Hub projects only. Six matching phrases (case-insensitive, whitespace-tolerant): `audit spoke projects`, `audit the spokes`, `which spokes are stale`, `show me spoke staleness`, `spoke inventory audit`, `run spoke audit`. The audit handler validates `topology.role == "hub"` and refuses on any other role with the verbatim message "Audit trigger valid only in Hub projects." On Hub projects, the handler reads the spoke inventory, compares each spoke's `Source Hub Version` against the version parsed from the attached `ai-engineering-hub-instructions-v*.md` file, sorts STALE alphabetical followed by current alphabetical, and emits a staleness report block. The audit trigger is read-only: it does not modify project files. Implementation: `references/preflight.md` section 12; semantics: `references/topology.md` section 4.
- **Multi-platform configuration loading.** New file `references/platform-specific-parameters.md` ships inside the .skill archive (not in operator project knowledge) with capability and limit parameters for three platforms: `claude-ai` (full values), `chatgpt-enterprise` (basic values), `m365-copilot` (tbd values pending platform validation). v0.6.0 reads but does not action this file; the project-creator skill (Workstream 3, week 2+) is the primary consumer.
- **`user-config.md.template` and `org-config.md.template` (new files, .template suffix).** Schema 0.4 templates for the user and org configuration layers, with operator placeholders (`[tbd]`) for required fields and skill defaults inline elsewhere. The schema is formalized in design spec §4.4 (user) and §4.5 (org). The skill auto-creates the rendered files in operator projects on first invocation if absent. Rendered files do not inherit the SPDX header from the template.
- **Scenario F migration path (v0.5.0 to v0.6.0).** New pre-flight scenario. Detects canonical files at schema "0.3" with `_managed_by: project-context-skill` and no `topology` block. Verdict: `⚠ Upgrade Available (v0.5.0 to v0.6.0)`. Operator confirmation token: `confirm v0.6.0 upgrade`. Migration rewrites the three files adding a `topology` block with `role: "unclassified"` default and bumping `schema_version: "0.3"` → `"0.4"`. After the upgrade, the skill emits the role-declaration prompt (LOCKED TEXT 1 per `references/preflight.md` section 13.1) to solicit the operator's topology role declaration. Algorithm: `references/migration.md` section 10.
- **Two new informational verdicts** in pre-flight: `⚠ Upgrade Available (v0.5.0 to v0.6.0)` (gates Scenario F upgrade) and `⚠ Stale Spoke` (informational; not blocking; fires when a spoke project's `topology.hub_version` does not match the version parsed from the attached Hub instructions file). Two adjacent edge-case verdicts also added: `⚠ Hub Source Behind` (rare; Hub file version older than spoke's declared version) and `⚠ Hub Source Missing` (no Hub instructions file attached). All four are informational; none block operation. Definitions: `references/preflight.md` section 4.2.
- **Topology validation at pre-flight** for schema 0.4 files. Validates `role` is one of five enum values, `declared_by` is `operator` or `skill-default`, `declared_at` parses as ISO 8601, and required-by-role fields are populated per `references/topology.md` section 6.1. Empty fields produce a Parse Error with message "Topology field `<field>` is empty; expected confident value, explicit placeholder, or null with reason." Implementation: `references/preflight.md` section 10.
- **Stale-spoke detection** on `role: spoke-*` projects. Compares `topology.hub_version` against the version parsed from an attached `ai-engineering-hub-instructions-v*.md` filename. Emits one of three informational verdicts (Stale Spoke, Hub Source Behind, Hub Source Missing). Implementation: `references/preflight.md` section 11.
- **Role-declaration prompts (LOCKED TEXT 1 and 2)** in `references/preflight.md` section 13. LOCKED TEXT 1 solicits a topology role on fresh projects (Scenario A) and after Scenario F upgrades. LOCKED TEXT 2 fires when the operator declares `role: spoke-*` but does not provide `hub_reference` and `hub_version`. The skill never writes a partial spoke topology; LOCKED TEXT 2 fires until both fields are provided.
- **Schema 0.4** with new REQUIRED `topology` block in frontmatter of all three files. The block is universal across roles; each role has its own required-non-null and required-null field set per `references/topology.md` section 6.1. Schema bump 0.3 → 0.4. `references/schema-changelog.md` adds the schema 0.4 entry with field-level diff and migration path.
- **`## Spoke Inventory` body section** (Hub projects only) in `project-context.md`. Markdown table with columns Name, Role, Artifact Type, Source Hub Version, Status, Parent. Operator-maintained; the skill does not auto-discover spokes across tenants in v0.6.0.

### Notes
- **Hub-spoke governance is the v0.6.0 first principle (P1).** A project's topology relationship to other projects is part of its active state. v0.5.0 and earlier described what a project was about; v0.6.0 also describes how it relates to other projects. The 2026-05-21 hub-spoke governance workshop locked the federal-state model and the two-layer supremacy mechanism; v0.6.0 implements the project-context portion (Workstream 1). The project-creator skill (Workstream 3) consumes the topology block, the spoke inventory, and `platform-specific-parameters.md` to render platform-appropriate skill instructions.
- **Multi-platform configuration deferred to project-creator.** v0.6.0 ships the platform-specific-parameters.md schema and the three platform entries (claude-ai full, chatgpt-enterprise basic, m365-copilot tbd) but does not action them. The project-creator skill (Workstream 3, week 2+) is the primary consumer. M365 Copilot platform parameters remain `tbd` pending a dedicated research pass.
- **Audit trigger is read-only.** The audit produces a staleness report but does not modify project files. Write-back (refreshing the spoke inventory's Status column in `project-context.md`) is deferred to a future trigger (`refresh spoke inventory`, out of scope for v0.6.0). Spoke upgrade (changing a spoke's declared Hub version and re-rendering its instructions) is the project-creator skill's upgrade mode, also out of scope for project-context.
- **Severity classification deferred.** v0.6.0 reports drift in the audit and stale-spoke detection without severity classification. A spoke 4 minor versions behind and a spoke 1 minor version behind both render as `⚠ Stale Spoke`. The operator decides priority. Future releases may add severity thresholds; v0.6.0 does not pre-empt the design.
- **Known untested inheritance from v0.5.0** (design spec §9.1 and §7.4): the forward-version refusal path (T5) and the v0.4 to v0.5 upgrade migration (T6) were not field-tested in the v0.5.0 cycle. v0.6.0's schema bump (0.3 → 0.4) exercises the T5 logic for any operator running v0.5.0 against a v0.6.0-upgraded project. Scenario F is structurally analogous to Scenario E (one additional schema step). Acceptance testing in Phase 3 of the v0.6.0 build exercises both Scenario E and Scenario F end-to-end.
- **Phase 1 + Patch 1 + Phase 2 build sequence.** v0.6.0 ships as a four-phase build: Phase 1 added the new reference files and SKILL.md surface updates (commit a49096f); Patch 1 corrected Codex Pass 1 findings (commit e5d80f5); Phase 2 added the pre-flight logic, migration paths, schema-changelog, SKILL.md body cleanup, and CHANGELOG.md entry (this entry); Phase 3 will run acceptance testing (16 acceptance tests plus forward-grounding regression check) and finalize the release.

## [0.5.0] — 2026-05-19

### Added
- **Mandatory pre-flight protocol (`## Protocol` section in SKILL.md).** Pre-flight is now a structural gate: SKILL.md's `## Protocol` is the first content after frontmatter, the model cannot reach operational content without passing through it, and the model must emit a pre-flight report block as the first content of every response before any generation. The protocol cites `references/preflight.md` as the authoritative source. Failure to complete pre-flight before generation is a protocol violation, not an optimization. Anti-rationalization clause is model-facing: operator urgency or perceived urgency does not license skipping pre-flight.
- **`references/preflight.md` (new file).** Authoritative specification for the pre-flight protocol and the symmetric post-flight summary. Contains: purpose (closing the postmortem-documented enforcement gap), first principle (project state is authoritative), pre-flight algorithm (three-tier `project_knowledge_search` strategy + six-step classification), pre-flight report block format with verdict glyphs `✓ ⚠ ✗`, scenario examples (A Fresh, B Matched, C Skill-too-old, D Legacy, E v0.4.0 Upgrade), confirmation token catalog, token matching rules, token mismatch error format, pre-flight completion criteria, infrastructure-failure handling, and post-flight summary specification.
- **Schema "0.3" with new REQUIRED `_managed_by: project-context-skill` field.** Frontmatter of all three files (`project-context.md`, `entities.md`, `project-context-archive.md`) carries this field. The leading underscore signals internal metadata, not user-facing data. The field is the registry marker that makes pre-flight detection reliable: `project_knowledge_search` for `_managed_by: project-context-skill` returns chunks where this distinctive YAML field actually appears, which is only in files under skill management. Schema bump 0.2 → 0.3. `references/schema-changelog.md` documents the field-level diff and the new "Supported Schemas" matrix.
- **Upgrade migration (v0.4.0 → v0.5.0).** New migration path: pre-flight detects canonical filenames with `schema_version: "0.2"` and no `_managed_by`, emits verdict `⚠ Upgrade Available`, and after `confirm upgrade` rewrites the three files adding `_managed_by` and bumping `schema_version` to `"0.3"`. All record content preserved verbatim — frontmatter-only modification. See `references/migration.md` section 9. Operator brief uses the download-and-replace pattern (no legacy-file deletion because filenames are unchanged).
- **Post-flight summary block.** Symmetric closing block to pre-flight: same verdict-glyph convention, same skippability-is-a-violation rule. Reports what was actually written, schema version, skill version, operation performed (with deviation note if execution diverged from pre-flight's proposal), and operator action required. Failure cases emit `✗ Failed` post-flight with diagnostic. Specification in `references/preflight.md` section 9.
- **Pre-flight prerequisite gating notes** at the top of each `operations/*.md` file (`default.md`, `merge_external.md`, `compact.md`, `rebuild.md`). Redundancy is intentional: the model encounters the gating signal both at SKILL.md entry and at each operation file's entry, so operations cannot be misread as standalone instructions.
- **Two new pre-flight verdicts:** `⚠ Upgrade Available` (v0.4.0 → v0.5.0 upgrade scenario) and `✗ Infrastructure Failure` (`project_knowledge_search` errored). Infrastructure failure does NOT default to assuming-fresh; the project may have state we cannot see.
- **"Supported Schemas" section** in `references/schema-changelog.md`. Authoritative compatibility matrix listing what v0.5.0 reads, writes, migrates from, and refuses. Pre-flight cites this matrix when classifying project state.
- **`USAGE.md` "Upgrading from v0.4.0" section.** Operator-facing walkthrough of the upgrade migration scenario, including the `confirm upgrade` token and the no-legacy-deletion flow.

### Migration

  - Upgrade migration (schema "0.2" → schema "0.3"): NEW. Frontmatter-only rewrite. Operator confirmation token: `confirm upgrade`. Brief uses the download-and-replace pattern (no legacy deletion).

## [0.4.0] — 2026-05-18

### Added
- **Three-file rolling architecture.** Output is now three files with rolling filenames: `project-context.md` (active grounding file), `entities.md` (stable reference data), `project-context-archive.md` (append-only history). Date lives in the `last_merged` frontmatter, not in the filename. Files are eagerly created on first invocation with HTML-comment-delimited placeholder blocks that the skill removes on first record write.
- **Four named operations** replacing the two modes. `default` (parse conversation, classify, propose, write), `merge_external` (same flow on an attached file), `compact` (batch-demote weak records), `rebuild` (regenerate active file from archive; mandatory pre-commit review). See `operations/`.
- **Five-op merge classifier.** ADD, UPDATE, NOOP, DEMOTE, SUPERSEDE — Mem0-style four plus the operator's DEMOTE extension. Classifier pseudocode lives in `references/operations.md`.
- **Hybrid brake** (default `merge_policy`). Auto-apply ADD and NOOP; gate UPDATE, DEMOTE, SUPERSEDE for operator approval. Alternative policies: `gate` (require approval on every change), `auto` (auto-approve everything, with mandatory warning and full audit trail).
- **Update-based scoring algorithm** with 6 tunable coefficients (`alpha`, `beta`, `gamma`, `delta`, `epsilon`, `lambda`) and a default `demotion_threshold` of 5. Decay is measured in file updates, not calendar time — a project untouched for six months sees zero decay. Formula and worked examples in `references/scoring.md`.
- **Auto-mode** shipped as a published option. Mandatory per-session warning matches design spec §12.3 verbatim. Every auto-approved record carries `audit.approval_mode: auto` and a visible `[AUTO]` prefix on `content`. Audit metadata enables coaching and quality diagnosis after the fact.
- **Per-record audit block.** Every record has `audit.approval_mode`, `audit.approved_by` (nullable in v0.4.0), `audit.approved_at`, `audit.warning_response`, `audit.importance_source`.
- **Three-layer configuration.** `user-config.md` (new in v0.4.0) > `org-config.md` > skill defaults from `references/defaults.md`. The `user-config.md` convention is intended as cross-skill: future skills in the monorepo adopt the same pattern. See `references/user-config-template.md`.
- **Decoupled schema versioning** plus `references/schema-changelog.md`. Schema is now `"0.2"` (the short, quoted form). The build session enforces a drift-detection guard against the prior tag.
- **`id_prefix_legend`** REQUIRED frontmatter map carried in every file the skill writes. Eight ID prefixes: `dec`, `con`, `csn`, `opn`, `trm`, `ref`, `ent`, `arc`.
- **Archive `checkpoints` array** in YAML frontmatter (not body). Per-merge log of what changed each update.
- **Migration** from v0.1.x-v0.3.x dated files. Automated, one-time, non-destructive. Operator brief lists each legacy file by exact filename with required order of operations (download new → verify → delete old → upload new). See `references/migration.md`.
- **ROADMAP.md** and **USAGE.md** added to the per-skill file set.

### Changed
- **Body sections** for the active file: was 7 (Decisions, Constraints, Entities, Terminology, External references, Open items, State snapshot) → now 6 (Decisions, Constraints, Current State, Open Items, Terminology, External References). "Entities" moved to its own file. "State snapshot" renamed "Current State."
- **Per-record format** rewritten. The v0.1 inline-bracket metadata (`[tier: full | summary | transient] [categories: tag1, tag2]`) is gone. Records now carry structured per-record YAML with lifecycle, scoring, status, provenance, links, and audit blocks.

- **Filename convention.** Dated filenames retired in favor of rolling filenames. The date now lives in `last_merged` frontmatter.

## [0.3.1] — 2026-05-11

### Fixed
- `modes/consolidate.md` step 9 validation invariant.
